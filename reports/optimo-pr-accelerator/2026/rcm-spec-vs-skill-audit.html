How Libraries Are Loaded — The Skill's Runtime Integration

Unlike the OptimoPR Edge Function (which has zero library integration), the skill's session protocol explicitly instructs Claude to load libraries:

# From references/rcm-session-protocol.md:
"For each perspective:
- State the perspective name and its lens
- Load relevant knowledge library sections
- Generate structured output in the perspective's domain
- Append to Project State under the relevant dimension(s)"

# From SKILL.md:
"- references/libraries/ — installed knowledge content: 8 base libraries + 4 engines"
"- references/knowledge-libraries.md — library catalog, domain-pack composition, install status"

The skill relies on Claude's in-context file reading — it instructs Claude to read the relevant .md files from the libraries directory and load them into the perspective prompts. This is the correct architecture per spec §9 and §11. The Library Access Matrix (§11) is implemented via the domain pack manifests which define which libraries each domain loads.

Architectural Divergences

Spec RequirementSkill ImplementationGap
Recursive loop (§1, §3): RCM Core → Project State → loop back. Continuous refinement.The session protocol runs 5 perspectives once (Dispatch → Synthesize → Next Actions). Recursion is offered as a next action: "Branch — fork the Project State to explore an alternative direction" and "Deep-dive — re-enter a specific workspace." The loop is human-mediated, not automatic.PARTIAL — recursion exists as manual next action, not automatic refinement loop
Stage Workspaces (§7): 5 sequential stages (Exploration→Evaluation→Execution→Risk/Safety→Social). Each stage runs all 5 agents but with ONE primary.The session protocol runs all 5 in one pass. The RCM session protocol references workspaces ("Exploration is default") but the dispatch phase runs all perspectives at once rather than as 5 sequential workspace stages with different primaries.PARTIAL — perspectives run, but not as formal workspace stages
Cross-Agent Review (§8): Every object reviewed by all 5 agents before Project State updates.The sequential pipeline (Explorer→Evaluator→Executor→Threat→Social, each reading prior output) provides implicit review. Evaluator reviews Explorer's concepts. But Threat and Social review the profile directly, not the other agents' specific concept objects.PARTIAL — sequential pipeline gives partial cross-review
Branching System (§15): Every concept may fork. Compare, Merge, Promote, Archive.The Project State schema has branch, parentBranch, and branchHistory[] fields. The session protocol offers "Branch — fork the Project State to explore an alternative direction." But no automated comparison/merge/promote tooling exists.PARTIAL — schema supports it, session offers it, but no automated tooling
